As opposed to the UtilsPlugins's abstract plugins, the general purpose plugins introduced here are fully working plugins.

In order to use them in your new amazing app, you just need to reference them from your application object, as stated in the application section.

tree

The tree is a navigation plugin.

It consists out of a beautifully crafted d3.js tree.
The root of the tree is the root of the navigation tree - Navigation.currentNavigationModel().
The navigation model's nodes ids are displayed in the tree.

There's a lot of heavy-weight functionality in this plugin, and explaining the code is more complicated than just looking at it...

One thing to notice is that the plugin's implementation has to keep information about each node. To do so, the plugin stores the info in the nodes themselves.

This isn't a recommended thing to do, but if you must make the navigation node objects dirty, make sure you do it in your own namespace (the tree uses the tree namespace - the info about a node is stored in node.tree).

table

The table is a navigation plugin.

It displays the navigation model in a tabular format: each row displays data about a single node.

The model has a tree structure, so this plugin only shows the kids of a single node.
The user can drill into the model - by double clicking a node in the table, its kids will now be displayed.

This plugin uses the application's getVisualization function in order to display extended info about each node (this data is displayed in each row's cells).
In addition, this plugin uses pagination - if there are too many kids, only a part of them will be displayed to the user.

One should use this plugin whenever the navigation tree hierarchy is too big and relatively flat. In this situation the tree fails for two reasons

  • The UI rendering might get stuck, as there are too many elements to be drawn in the tree.
  • Even if doesn't get stuck, the UX is bad: the user will get a hard time navigating through the tree.

The table plugin solves this problem by splitting a node's kids to pages, and only one page is visible at a time.

As with the tree case, in order to understand the table plugin, it's much simpler to read the code itself.

circles

The circles is a navigation plugin.

In order to use it, the items of the navigation nodes should have a size field (or some of the items at least, as described below).

This plugin gives a different navigation UX. the nodes hierarchy is represented as nested circles.

The size of each circle is proportional to the size of the node's item. If the size isn't defined, it will sum the sizes of the node's kids, and set it as the new size (this is a recursive process).

This visualisation gives a feeling of the interactions between the nodes, e.g: bigger circle might be more important than smaller ones.

The application constructing the navigation model should put into the size field something which makes sense to the application.
